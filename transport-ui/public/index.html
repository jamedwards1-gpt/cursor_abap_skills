ABAP “terminal” and HTTP

      

        There is no ADT REST equivalent of the Eclipse ABAP terminal for arbitrary statements on BTP ABAP Environment.
        The ADT HTTP APIs are things like object CRUD, CTS, search, activation — not a REPL. The @mcp-abap-adt/adt-clients “SQL Data Preview”
        helper is documented as blocked for JWT/XSUAA on ABAP Cloud, so we do not expose a SQL runner here.
      

      

        First-principles workaround: run logic inside the system as an
        IF_HTTP_SERVICE_EXTENSION handler (you already do this with ZCL_PARCEL_POLL_UI_HTTP), publish it as an HTTP service,
        then call it with the same identity as ADT (JWT in .secrets/btp-abap.env) using the safe GET proxy below — path only, allowlisted prefixes.
      

      
Example class in this repo

      

        ZCL_TRANSPORT_UI_STATIC_JSON (btp-content/abap/ZPARCEL/CLAS_OC/…) returns a tiny JSON document on GET.
        Push it to Steampunk, create an HTTP Service in ADT that references the class, publish, then paste the path part of the published URL here
        (must start with /sap/bc/, /sap/opu/, or /default_host/).

All my transports

      

        Uses ADT GET …/cts/transportrequests?user=… for your BTP user id (e.g. CB9980000010), then loads each request.
        Leave User empty or use Use env user so we take the id from JWT / SAP_USERNAME — partial values like C are replaced automatically.
        Status is only a short ADT code such as D (modifiable); do not paste your user id or a transport number there.
      

      

        

        

        

        

        

      

      
      
      
      

      

    


    

      
Mini STMS — catalog & basket

      

        Search uses ADT quick search; package index uses package explorer. Add to basket then assign with a task number (modifiable task).
        Supported for assign: CLAS/*, INTF/*, PROG/P, TABL/*, DDLS/* — lock with corrNr then unlock (same pattern as ADT recording).
